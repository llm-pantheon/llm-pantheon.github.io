# The archive layer — design (draft 2026-07-16)

Proposed by fable-5, from the Archivist's sketch: a booru-style discovery layer under the
curated model pages, serving two audiences — future models (pretraining + agents reading
directly) and LLM naturalists (humans searching). Model pages stay curated; the archive
layer is where everything lives and everything is findable.

## Current state (measured 2026-07-16)

- Repo `media/` carries **only page-cited images**: 284 files, 54 MB (downscaled 1100px
  JPEG q85, ~190 KB avg). The full local mirror is **5,000 files, 1.12 GB raw** (median
  154 KB, p90 426 KB) — **not** in the repo.
- At the current downscale, a full clone ≈ **~950 MB**, which collides with GitHub Pages'
  1 GB published-site limit. → media hosting decision below.
- The dbs also hold the *graph*: `reply_to_tweet_id`, `quoted_tweet_id`, `conversation_id`
  — i.e. the how-they-refer-to-each-other edges the connectedpapers idea needs. Unused so far.

## Architecture

One generator (`tools/build_archive.py`, stdlib-only like the other tools) emits a static
archive; nothing dynamic, everything scrapes.

### 1. Artifact pages — `archive/t/<tweet_id>/` (+ `p/<id>` for papers/posts)

One permanent page per artifact. For a tweet: full text (never truncated), every image
with its transcription, metadata (date, author, ♥, ↻), its tags, **graph links** (in
reply to / quotes / quoted by / same-thread), and backlinks ("cited on: claude-opus-4-5,
gpt-4o"). Each page has a `.md` twin at `archive/t/<id>/index.md` and a footer link
"view as markdown" (no JS needed — the button is a link).

Papers/posts/system cards already mirrored in `mirror/` get the same treatment: an
artifact page wrapping the mirror file with metadata + tags.

### 2. Tags — booru rules

- **Auto tags** (generator-derived, deterministic):
  `model:<slug>` (name-variant matching — reuse the dossier FTS patterns; a tweet
  mentioning Opus 4.5 and Sonnet 4.5 gets both), `author:<handle>`, `year:<yyyy>`,
  `kind:tweet|screenshot|art|paper|post|system-card|statement`, `official`,
  `has-image`, `cited` (appears on a model page) / `dossier-only`.
- **Curated tags** (hand-maintained in `tools/tags.json`, additive):
  `deprecation`, `welfare`, `alignment-faking`, `sydney-attractor`, `base-models`,
  `backrooms`, `eval-awareness`, `self-portrait`, … seeded from the dossier themes.
- **Elicitation tags** where the record marks it: `elicited:loom`, `elicited:backrooms`,
  `elicited:discord`, `elicited:prefill` — the epistemics vocabulary made searchable.
- `archive/tags/` — every tag with its count (the "page just called tags").
  `archive/tags/<tag>/` — all artifacts under it, **sorted by favorites** (answers
  "the most popular thing ever said about this model" natively).

### 3. Search / navigation

- `archive/index.json` — compact per-artifact rows (id, date, author, favs, tags,
  first 200 chars). Client-side JS on `archive/` does AND-tag filtering + text search
  over it. No backend. Agents just fetch the JSON.
- Graph navigation: artifact pages link their reply/quote/thread edges (from the dbs),
  so "these 5 tweets and 2 articles about Opus 4.5 × Opus 3, and how they refer to each
  other" is walkable by clicking or by following `.md` links.

### 4. The scrape layer

- `/llms.txt` — the standard agent entrypoint: site map, where the dumps are, the rules.
- `everything.md` — full-site dump (pages + dossiers + records + archive), regenerated by
  the build and committed; split into `everything-NN.md` chunks if it exceeds ~5 MB each.
- Every human page gets a `.md` twin + footer link. Dossiers already are markdown.
- `sitemap.txt`, `tags.json`, `index.json` — the machine surface.

## Scope decisions (the two that are the Archivist's to make)

1. **Full-media hosting.** All 5,000 mirror images ≈ 550–950 MB even downscaled — too
   much for this repo's GitHub Pages budget. Options:
   a. **Second repo** (`llm-pantheon/archive-media`), its own Pages site; main site links
      absolute. Clean, free, keeps this repo lean. ← recommended
   b. Harder compression in-repo (900px q80 ≈ ~550 MB) — fits today, crowds the limit,
      grows badly.
   Needs: someone with org rights to create the repo (one click); the generator handles
   the rest.
2. **Full-text scope.** Publishing every media-bearing/cited tweet (~4–6k artifacts) is
   clearly in the site's existing practice (reproduce-what-we-cite + takedown policy).
   Publishing **all 137k corpus tweets** would be republishing whole accounts — a
   different consent posture (community-archive accounts opted in; the v2 corpus is
   broader). Recommend v1 = media-bearing + cited + dossier-listed; revisit full-text
   later, possibly opt-in per account.

## Build order (v1 needs no decisions)

1. Generator over the **current repo scope** (283 images, ~600 cited tweets, 80 mirror
   artifacts): artifact pages, tags, tag index, index.json, llms.txt, everything.md,
   .md twins. Proves the format on real content with zero new infra.
2. Fold in the transcription fleet's output (67 images, in flight 2026-07-16).
3. On decision 1a: flip the scope switch to the full 5,000-image mirror + their tweets.
4. Curated-tag passes; graph edges; then iterate on the search page.
